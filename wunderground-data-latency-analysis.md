# Wunderground 数据源延迟分析报告

> **项目背景**：Polymarket 温度预测市场  
> **分析日期**：2026-04-07  
> **目标站点**：LEMD — Adolfo Suárez Madrid-Barajas Airport Station  

---

## 一、需求描述

### 1.1 业务场景

在 Polymarket 上参与温度预测市场（如 [Highest temperature in Madrid on April 7?](https://polymarket.com/zh/event/highest-temperature-in-madrid-on-april-7-2026)），需要尽可能快速地获取 Wunderground 数据源的温度数据，以便在数据公开后第一时间做出交易决策。

### 1.2 市场规则要点

- **结算数据源**：Weather Underground 的 [LEMD 历史页面](https://www.wunderground.com/history/daily/es/madrid/LEMD)
- **精度**：整数摄氏度（如 9°C）
- **结算时机**：当天所有数据 **finalized** 之后才能 resolve
- **不可追溯**：数据最终化后的修订不影响结算

### 1.3 核心问题

1. Wunderground 页面数据从哪里来？更新频率是多少？
2. 能否通过 API 直接获取，比网页更快？
3. 服务器部署在哪里？是否有 CDN 缓存延迟？
4. 是否有更上游的数据源可以提前获得信号？

---

## 二、解决思路

### 2.1 逆向分析 Wunderground 前端

通过对 Wunderground 历史页面（`/history/daily/es/madrid/LEMD`）进行 HTML 源码分析和网络请求抓包，还原其前端调用的后端 API 端点及认证方式。

### 2.2 直接测试 API 性能

绕过网页前端，直接调用底层 `api.weather.com` 的 REST API，测量响应时间和缓存行为。

### 2.3 延迟测试

编写自动化轮询脚本，在 METAR 观测时间点（整点/半点）前后密集轮询两套 API，精确测量"气象站观测 → API 可查询"的延迟。

### 2.4 基础设施调研

调查 Wunderground/The Weather Company 的服务器架构、CDN 分布，评估就近部署服务器的可行性。

### 3.2 API 响应验证

#### v3 实时观测 API

```bash
curl 'https://api.weather.com/v3/wx/observations/current?apiKey=e1f10a1e78da46f5b10a1e78da96f525&units=m&format=json&icaoCode=LEMD'
```

返回结果（截取关键字段）：
```json
{
    "temperature": 13,
    "temperatureMax24Hour": 26,
    "temperatureMaxSince7Am": 26,
    "temperatureMin24Hour": 9,
    "validTimeLocal": "2026-04-07T04:26:33+0200",
    "validTimeUtc": 1775528793
}
```

**关键发现**：
- `temperatureMaxSince7Am` 字段直接给出当天 7AM 以来的最高温度，这正是 Polymarket 市场关心的值
- `max-age=26` 意味着缓存仅 26 秒，近乎实时

#### v1 历史观测 API

```bash
curl 'https://api.weather.com/v1/location/LEMD:9:ES/observations/historical.json?apiKey=e1f10a1e78da46f5b10a1e78da96f525&units=m&startDate=20260407&endDate=20260407'
```

返回当天所有逐次观测，每 30 分钟一条记录：
```
22:00 UTC | 17°C
22:30 UTC | 17°C
23:00 UTC | 16°C
23:30 UTC | 16°C
00:00 UTC | 14°C
00:30 UTC | 15°C
01:00 UTC | 14°C
01:30 UTC | 14°C
02:00 UTC | 13°C
```

**关键发现**：
- 观测间隔为 30 分钟，与 METAR 报文周期一致
- `s-maxage=3600`：CDN 缓存 1 小时，意味着新的 METAR 观测可能最多延迟 1 小时才出现在此 API 中
- 这就是 Wunderground 历史页面表格的直接数据源

### 3.3 HTTP 响应头深度分析

| API | Cache-Control | 实际含义 |
|-----|---------------|----------|
| v3 实时观测 | `max-age=26` | 浏览器/客户端缓存 26 秒 |
| v1 历史观测 | `public, max-age=2959, s-maxage=3600` | CDN 缓存 1 小时，客户端 ~50 分钟 |
| Wunderground HTML 页面 | `max-age=0, no-cache` | 页面本身不缓存（SSR），但后端 API 有缓存 |

所有响应都带有 `akamai-grn` 头，确认经过 Akamai CDN。

### 3.5 延迟实测

#### 测试方法

编写 Python 脚本 `wu_latency_test.py`，同时轮询 v3 和 v1 两个 API，在 METAR 时间点前后密集轮询（5-15 秒间隔），检测观测数据变化并记录延迟。

#### 测试结果

**测试时间**：2026-04-07 03:03 ~ 03:07 UTC

| 事件 | 时间 | 详情 |
|------|------|------|
| 初始状态 | 03:04:14 UTC | v3 最新观测 02:56:40 UTC (13°C)<br>v1 最新观测 02:00:00 UTC (13°C, 共 9 条) |
| v3 检测到新观测 | 03:06:46 UTC | 观测时间 03:06:45 UTC → **延迟 1 秒** |
| v3 检测到新观测（第二实例） | 03:06:51 UTC | 同一观测 03:06:45 UTC → **延迟 6 秒** |
| v1 状态 | 03:07:53 UTC | 仍为 02:00 UTC，无变化（CDN 缓存未过期） |

#### 关键发现

```
┌─────────────────────────────────────────────────────┐
│  v3 实时观测 API 延迟：1 ~ 6 秒                      │
│  v1 历史观测 API 延迟：最长可达 1 小时（CDN 缓存）     │
│  差距：v3 比 v1 快 ~60 倍                             │
└─────────────────────────────────────────────────────┘
```

- **v3 实时 API 的数据几乎实时可用**（1-6 秒延迟），远远快于 Wunderground 网页展示的数据
- **v1 历史 API 受 CDN 1 小时缓存限制**，即使底层数据已更新，API 也可能返回旧数据
- v3 API 的观测时间（02:56:40、03:06:45）不在 :00/:30 整点，因为 v3 使用的是 AWOS 高频数据流而非 METAR 报文（详见 §6.4）
- HTTP 响应时间约 640-1350ms（从本地 macOS 测试，非欧洲服务器）

## 四、结论与建议

### 4.1 数据获取最优方案

```
推荐方案：v3 实时观测 API（26 秒缓存，1-6 秒延迟）
```

直接轮询 `api.weather.com/v3/wx/observations/current?icaoCode=LEMD`，每 30 秒一次，跟踪 `temperatureMaxSince7Am` 字段。这比 Wunderground 网页快最多 **1 小时**。

示例请求：
```
GET https://api.weather.com/v3/wx/observations/current
    ?apiKey=e1f10a1e78da46f5b10a1e78da96f525
    &language=en-US
    &units=m
    &format=json
    &icaoCode=LEMD
```

关键返回字段：
- `temperature` — 当前温度
- `temperatureMaxSince7Am` — 当天 7AM 起最高温度（**市场结算的关键值**）
- `temperatureMax24Hour` — 过去 24 小时最高温度
- `validTimeUtc` — 观测时间戳

### 4.2 关于服务器部署位置

| 因素 | 分析 |
|------|------|
| 当前延迟瓶颈 | CDN 缓存（1 小时），不是网络延迟 |
| 网络 RTT 节省 | 从美国 ~500ms → 欧洲 ~20-50ms，节省 ~450ms |
| 相比 CDN 缓存 | 450ms vs 3600s，网络延迟占比 < 0.01% |
| **结论** | 如果用 v3 实时 API（无 CDN 缓存问题），部署位置不重要<br>如果用 v1 历史 API，部署位置同样不重要（瓶颈在缓存） |

**建议**：如果你已有欧洲服务器，可以用它来轮询。否则无需为此专门部署，因为 v3 API 的缓存仅 26 秒，网络延迟相对可忽略。

### 4.3 更上游数据源（信号领先）

除了 Wunderground，以下数据源可以更早获得温度信号：

| 数据源 | 预计领先时间 | 说明 |
|--------|-------------|------|
| **AEMET OpenData**（西班牙气象局） | 10-30 分钟 | 最上游，免费 API，需申请 Key |
| **NOAA/ADDS METAR** | 5-15 分钟 | 航空气象，`aviationweather.gov` |
| **OGIMET** | 5-15 分钟 | 原始 SYNOP/METAR 报文 |

注意：Polymarket 结算依据的是 Wunderground，不是这些上游源。但上游数据可以帮你**提前预判最终结果**并更早下单。

### 4.4 关于"数据最终化"时机

Polymarket 规则要求"data finalized"后才能结算。根据分析：

- Wunderground 历史页面的数据来自 v1 历史 API
- 当天的最后一条 METAR 观测通常在马德里时间午夜（UTC 22:00）之后
- 数据最终化大约在次日 UTC 00:00 ~ 06:00 之间
- v1 API 的 `metadata.expire_time_gmt` 字段可以用来追踪缓存过期时间

### 4.5 完整数据架构建议

```
层级 1: 信号源（提前获知温度）
├── AEMET OpenData API（最上游）
├── NOAA ADDS METAR
└── WU v3 实时观测 API（26s 缓存，1-6s 延迟）
    └── 持续追踪 temperatureMaxSince7Am

层级 2: 结算数据源（确认最终结果）
└── WU v1 历史观测 API（1h CDN 缓存）
    └── 轮询检测数据最终化
    └── 最终化后与层级 1 交叉验证

层级 3: 交易执行
└── 已有 Polymarket 交易系统
    └── 触发买入/卖出
```

## 五、风险提示

1. **API Key 风险**：前端内嵌 Key 非官方公开，可能被更换或限流
2. **缓存一致性**：v3 API 的 `temperatureMaxSince7Am` 与 v1 历史 API 的最终最高温可能存在舍入差异
3. **METAR 修订**：气象站可能发布修订后的 METAR（METAR COR），Wunderground 是否采纳存在不确定性
4. **频率限制**：高频轮询可能触发 IP 限流或封禁，建议控制在每 30 秒一次

## 六、WU 数据源与 METAR 关系深度分析

### 6.1 数据溯源：WU 的温度数据从哪来？

Weather Underground 是 **The Weather Company**（TWC，IBM 旗下）的消费端品牌。对于 LEMD（Madrid-Barajas Airport）等 ICAO 注册站点，WU 展示的气温数据**本质上就是 METAR 报文中的温度值**。

完整的数据链路：

```
┌──────────────────────────────────────────────────────────┐
│  AEMET 气象站（马德里巴拉哈斯机场跑道旁）                   │
│  ├── 自动观测设备（AWOS/ASOS）实时采集温湿度/气压/风速      │
│  └── 气象员审核（有人值守时段）                             │
└────────────────────┬─────────────────────────────────────┘
                     ▼
┌──────────────────────────────────────────────────────────┐
│  METAR 报文编码（ICAO Annex 3 / WMO No. 49 标准）        │
│  示例：METAR LEMD 070300Z 36004KT 9999 FEW040 13/08     │
│        Q1020 NOSIG                                       │
│  └── 其中 "13/08" = 温度 13°C / 露点 8°C                 │
└────────────────────┬─────────────────────────────────────┘
                     ▼
┌──────────────────────────────────────────────────────────┐
│  GTS（全球电信系统）分发                                   │
│  ├── AEMET → ECMWF（欧洲中期天气预报中心）                 │
│  ├── AEMET → NOAA/NWS（美国国家气象局）                    │
│  ├── AEMET → 各国气象局                                   │
│  └── AEMET → 商业数据聚合商（含 The Weather Company）      │
└────────────────────┬─────────────────────────────────────┘
                     ▼
┌──────────────────────────────────────────────────────────┐
│  The Weather Company 数据管道                             │
│  ├── METAR 解析：编码文本 → 结构化数据                     │
│  ├── QC（质量控制）：异常值检测、站点一致性校验              │
│  ├── 增值计算：temperatureMaxSince7Am 等派生字段           │
│  └── 写入数据库 → 暴露为 REST API                         │
└────────────────────┬─────────────────────────────────────┘
                     ▼
┌──────────────────────────────────────────────────────────┐
│  api.weather.com（Akamai CDN 加速）                       │
│  ├── v3 实时 API（max-age=26s）  ← 推荐，延迟 1-6 秒      │
│  │   └── 数据源：AWOS 高频推送（~10 分钟间隔）             │
│  └── v1 历史 API（s-maxage=3600s）← WU 页面数据源          │
│      └── 数据源：METAR 报文（30 分钟间隔）                 │
└────────────────────┬─────────────────────────────────────┘
                     ▼
┌──────────────────────────────────────────────────────────┐
│  wunderground.com/history/daily/es/madrid/LEMD           │
│  ├── Next.js 服务端渲染（SSR）                             │
│  │   └── 服务端调用 v1 API → 数据不暴露在前端 JS 中        │
│  └── 渲染 "Daily Observations" 表格                       │
│      └── 48 行/天，每行 = v1 API 一条记录，无任何修改      │
└──────────────────────────────────────────────────────────┘
```

### 6.2 什么是 METAR？

**METAR**（METeorological Aerodrome Report）是国际民航组织（ICAO）制定的航空例行天气报告格式，全球所有民用机场必须按标准发布。

| 属性 | 说明 |
|------|------|
| **发布频率** | 通常每 30 分钟或 60 分钟一次（LEMD 为 30 分钟） |
| **发布时间** | 标称整点/:30，实际观测时间有 ±3-7 分钟偏差 |
| **温度精度** | 主体部分为整数°C；RMK（备注）部分可含 0.1°C 精度 |
| **覆盖要素** | 温度、露点、风向风速、能见度、云量、气压、天气现象 |
| **分发渠道** | GTS 全球分发，通常观测后 2-5 分钟内可在 NOAA ADDS 查到 |
| **特殊报文** | SPECI（特殊天气报告，阈值触发）、METAR COR（修正报文） |

#### METAR 报文示例解读

```
METAR LEMD 070300Z 36004KT 9999 FEW040 13/08 Q1020 NOSIG
│     │    │       │       │    │      │     │     │
│     │    │       │       │    │      │     │     └── 趋势：无显著变化
│     │    │       │       │    │      │     └── 修正海平面气压 1020 hPa
│     │    │       │       │    │      └── 温度 13°C / 露点 8°C
│     │    │       │       │    └── 少云，云底 4000 英尺
│     │    │       │       └── 能见度 10km+
│     │    │       └── 风向 360°（正北），风速 4 节
│     │    └── 7 日 03:00 UTC
│     └── 站点 ICAO 代码
└── 报文类型
```

### 6.3 从传感器到 METAR：自动观测系统深度解析

#### AWOS/ASOS vs SIM：各国系统对照

| 系统 | 全称 | 运营方 | 适用地区 |
|------|------|--------|---------|
| **AWOS** | Automated Weather Observing System | FAA（美国联邦航空局） | 美国 |
| **ASOS** | Automated Surface Observing System | NWS + FAA + DoD | 美国 |
| **SIM** | Sistema Integrado Meteorológico | **AEMET**（西班牙国家气象局） | **西班牙（含 LEMD）** |

LEMD 马德里机场使用的不是美国的 AWOS/ASOS，而是 AEMET 自主开发的 **SIM（综合气象系统）**。根据 [AEMET 官方文档](https://www.aemet.es/en/idi/observacion/observacion_aeronautica)：

> SIM 由部署在跑道接触区、中点、起飞区的多组自动观测站组成，连接至中央计算机，负责采集、处理并发送气象报告。系统能够**自动生成 METAR 和 SPECI 报文**。

LEMD 还额外配备了**风廓线仪和 RASS 系统**（Radio Acoustic Sounding System），提供连续的风和温度垂直剖面观测。

但无论叫 AWOS、ASOS 还是 SIM，**核心架构相同**——都是"传感器连续采集 → 计算机处理 → 定时生成标准报文"。

#### 传感器到 METAR 的完整数据流

```
层级 1: 物理传感器（连续采集）
┌──────────────────────────────────────────────────────────┐
│  跑道旁的传感器阵列（每 1-5 秒采样一次）                    │
│                                                          │
│  ├── 温度/露点探头     → 每秒采样，5 分钟滑动平均           │
│  ├── 风速/风向传感器   → 每秒采样，2 分钟滑动平均           │
│  ├── 气压传感器        → 每秒采样，即时值                   │
│  ├── 能见度仪(透射仪)  → 每秒采样，1-2 分钟有效窗口         │
│  ├── 云幕仪(激光测云)  → 连续扫描，30 分钟窗口双权重平均    │
│  └── 天气现象传感器    → 降水类型/强度，持续检测             │
└────────────────────┬─────────────────────────────────────┘
                     ▼
层级 2: SIM 中央处理计算机（实时处理）
┌──────────────────────────────────────────────────────────┐
│  每 ~1 分钟 → 生成内部观测记录（1-min observation）         │
│  │  └── TWC 可能通过商业协议获取此层数据 → v3 API           │
│  │                                                       │
│  每 30 分钟 → 触发 METAR 生成算法                          │
│  │  ├── 在 :00/:30 前约 10 分钟开始采集窗口                 │
│  │  ├── 对各要素应用不同的时间窗口算法                      │
│  │  ├── 编码为 METAR 格式                                  │
│  │  └── 气象员审核（有人值守时段）或直接发布（AUTO）         │
│  │                                                       │
│  随时 → 触发 SPECI（特殊报告）                             │
│     └── 当任一要素突破阈值（风切变、雷暴、能见度骤降等）     │
└────────────────────┬─────────────────────────────────────┘
                     ▼
层级 3: METAR 报文发布 → GTS 全球分发
└── NOAA、OGIMET、各国气象局、商业数据商均可接收
```

#### METAR 温度的生成算法：不是"瞬时读数"

METAR 报文中的温度**不是某一个瞬间的传感器读数**，而是经过时间窗口算法处理后的代表值。各气象要素使用不同的时间窗口：

| 气象要素 | 传感器采样率 | METAR 使用的时间窗口 | 处理算法 |
|---------|-----------|-------------------|---------|
| **温度/露点** | 每秒 | **最近 5 分钟** | 滑动平均/代表值，取整到 1°C |
| **风速/风向** | 每秒 | **最近 2 分钟**（平均风）；**10 分钟**（阵风） | 2 分钟矢量平均 |
| **能见度** | 每秒 | **最近 10 分钟** | 10 分钟平均 |
| **云量/云高** | 连续激光 | **最近 30 分钟** | 30 分钟窗口，最后 10 分钟双倍权重 |
| **气压** | 每秒 | **即时值** | 报文时刻的瞬时读数 |

示例：LEMD 在 03:00 UTC 的 METAR 温度（12°C）如何产生：

```
时间线 (UTC)：
  02:55:00 ── 传感器: 12.3°C  ┐
  02:56:00 ── 传感器: 12.1°C  │
  02:57:00 ── 传感器: 12.2°C  │ ← 5 分钟平均窗口
  02:58:00 ── 传感器: 12.3°C  │    (约 02:55~03:00)
  02:59:00 ── 传感器: 12.0°C  │
  03:00:00 ── 传感器: 11.9°C  ┘
                               → 平均 ≈ 12.13°C → 取整 → 12°C
                               → 编码为 "12/04"
                               → METAR LEMD 070300Z ... 12/04 Q1019 NOSIG
```

**这就是为什么 v3 API 和 METAR 温度通常一致**：温度短时变化缓慢（30 分钟内通常 < 1°C），5 分钟平均值和任意时刻瞬时值的差异很小，取整到 1°C 后差异消失。

#### 为什么 v3 API 能做到 ~10 分钟更新？

完整解释：

```
同一套传感器（跑道旁温度探头，每秒采样）
       │
       ├── 路径 A: SIM 内部 1-min 观测记录
       │   └── TWC 商业管道接收 → 约 10 分钟聚合一次 → v3 API
       │       时间戳如 02:56:40, 03:06:45（SIM 内部时钟）
       │       值 = 瞬时/短窗口平均值（未经 METAR 格式化）
       │
       └── 路径 B: METAR 生成算法（每 30 分钟）
           └── 5 分钟窗口平均 → 取整 → METAR 编码
               └── GTS 分发 → NOAA/OGIMET（延迟 2-5 分钟）
               └── TWC 接收 → v1 历史 API（CDN 缓存 ≤1 小时）
```

| 对比项 | v3 API（路径 A） | METAR / v1 API（路径 B） |
|--------|-----------------|------------------------|
| 数据来源 | SIM 内部 1-min obs | METAR 正式报文 |
| 更新频率 | ~10 分钟 | 30 分钟 |
| 温度处理 | 瞬时/短窗口值 | 5 分钟平均 + 取整 |
| 时间戳 | 非标准（秒级精度） | 标准 :00/:30 UTC |
| 数据延迟 | 1-6 秒 | 2 分钟~1 小时 |
| 温度差异 | 通常与 METAR 一致（<0.5°C 差异取整后消失） | 基准值 |

**7/7 天完全一致**。从 METAR 计算的日最高温 = WU v1 API 的日最高温 = v3 API 的 `temperatureMaxSince7Am`。

### 7.6 逆向推导的 METAR 温度生成算法

```
输入: 温度传感器连续读数 T(t)，采样率 ~1 秒

算法:
  1. 取最近 5 分钟（约 300 个采样点）的滑动平均:
     T_avg = mean(T(t-300s) ... T(t))

  2. 取整到最近整数°C:
     T_metar = round(T_avg)

  3. 每 30 分钟（:00 和 :30 UTC）输出一次

简化公式（实用近似）:
  METAR_temperature ≈ round(当前传感器温度)

误差来源:
  • 5 分钟平均 vs 瞬时值: 97.8% 情况下 < 0.5°C（取整后消失）
  • 仅在快速变化期（日出后升温/日落后降温）可能有 ±1°C 差异
```

### 7.8 v3 API → METAR 实时预测系统

#### 7.8.1 核心思路

v3 API 每 ~10 分钟从 SIM（AWOS）系统获取一次新的传感器读数，而 METAR 每 30 分钟才发布一次。这意味着在 METAR 发布前 ~10-20 分钟，v3 已经有了接近 METAR 时刻的温度值。利用这个时间差，可以**提前预测下一条 METAR 的温度**。

```
时间线示例 (METAR at :00 and :30 UTC):
  :00  METAR 发布 → v3 在 :02 左右更新（METAR 内容）
  :10  v3 新观测 ← 开始预测 :30 METAR
  :20  v3 新观测 ← 最佳预测时机（距 METAR 仅 10min）
  :30  METAR 发布 → v3 在 :32 左右更新
```

**实际信息优势 ≈ 15-25 分钟**：
- 10min：v3 在 METAR 前 10 分钟就有接近的传感器读数
- 5-15min：METAR 发布后，TWC 处理 + WU 页面更新的额外延迟

#### 7.8.2 预测模型

```
predicted_METAR = round(v3_temp + trend × Δt)

其中:
  v3_temp  = v3 API 最新温度 (整数 °C)
  trend    = 最近两次 v3 观测的变化率 (°C/min)
  Δt       = 距下一次 METAR 的分钟数 (:00 或 :30)
  round()  = 四舍五入到整数
```

#### 7.8.3 模拟回测（360 条 METAR，2026-03-31 ~ 04-07）

由于 v3 API 无历史端点，使用 METAR 历史数据通过线性插值模拟 v3 在不同提前时间的温度读数，推算预测准确率：

| v3 提前时间 | 预测准确率 | ±1°C 内 | 场景说明 |
|------------|----------|---------|---------|
| 30min | 44.8% | 90.5% | 最坏情况：无 v3，用上一条 METAR |
| 20min | 44.8% | 97.8% | v3 较早的观测 |
| 15min | 68.0% | 98.9% | v3 中间观测 |
| **10min** | **90.5%** | **99.7%** | **典型场景：v3 标准更新间隔** |
| 5min | 98.1% | 100.0% | 最佳情况：v3 恰好在 METAR 前 5min |
| 2min | 100.0% | 100.0% | 理想场景 |

**关键结论：v3 以 ~10 分钟间隔更新时，预测下一条 METAR 温度的准确率为 90.5%，且 99.7% 误差在 ±1°C 内**。

#### 7.8.4 30 分钟温度变化分布

```
变化 0°C: 161/359 (44.8%) — v3 读数直接 = 下一条 METAR，准确率 ~100%
变化 1°C: 164/359 (45.7%) — v3 以 ~10min 间隔可捕捉到趋势
变化≥2°C:  34/359 ( 9.5%) — 主要出现在日出后快速升温期 (7-10AM local)
```

#### 7.8.5 日最高温锁定策略（Polymarket 核心）

回测 8 天数据显示，日最高温通常在 14:00-16:00 UTC（16:00-18:00 Madrid 当地时间）达到：

| 日期 | 最高温 | 首次达到(UTC) | 确认下降(UTC) | 确认后到日终的时间 |
|------|--------|-------------|-------------|-----------------|
| 03-31 | 20°C | 14:00 | 14:30 | 9.0h |
| 04-01 | 19°C | 13:30 | 16:00 | 7.5h |
| 04-02 | 17°C | 13:30 | 17:30 | 6.0h |
| 04-03 | 22°C | 16:00 | 16:30 | 7.0h |
| 04-04 | 23°C | 15:00 | 18:00 | 5.5h |
| 04-05 | 26°C | 16:00 | 16:30 | 7.0h |
| 04-06 | 25°C | 14:00 | 17:30 | 6.0h |

> "确认下降"= 温度首次低于当日最高的 METAR 时刻（此时才能确认峰值已过）。
> "确认后到日终"= 从确认下降到 23:30 UTC 的剩余时间，即可用于交易的窗口。

**平均提前 6.9 小时确认当日最高温已锁定**。一旦 v3 的 `temperatureMaxSince7Am` 不再上升且当前温度开始下降，日最高温即已确定。

#### 7.8.7 Polymarket 交易决策树

```
┌────────────────────────────────────────────────────────────────────┐
│  Step 1: 监控 v3 API                                               │
│    v3.temperatureMaxSince7Am = M                                   │
│    v3.temperature = T (当前温度)                                    │
│    trend = 最近两次 v3 观测的变化方向                                │
│                                                                    │
│  Step 2: 判断当日最高温是否已锁定                                   │
│    IF T < M AND trend ≤ 0:                                         │
│      → 日最高温 = M（已锁定 ✅）                                    │
│      → 高置信度交易                                                 │
│    ELIF T == M AND trend > 0:                                      │
│      → 日最高温可能还会升 → 等待下一次 v3 更新                      │
│    ELIF T == M AND trend ≤ 0:                                      │
│      → 日最高温大概率 = M → 等 1-2 次 v3 确认                      │
│                                                                    │
│  Step 3: 与 Polymarket 市场对比                                     │
│    IF 市场赔率未反映已锁定的 M → 存在套利机会                       │
│    IF 市场已充分定价 → 无操作                                       │
└────────────────────────────────────────────────────────────────────┘
```

### 7.9 补充数据源：AEMET OpenData 10 分钟数据

AEMET（西班牙国家气象局）通过 OpenData API 提供 LEMD 站点的 **10 分钟观测数据**，这是最接近 SIM 内部原始数据的公开来源。

**获取方式**：
1. 注册 AEMET OpenData 账号：`https://opendata.aemet.es/centrodedescargas/altaUsuario`
2. 获取 API Key
3. 调用 10 分钟观测接口：`/api/observacion/convencional/datos/estacion/3129`

> 如需更精确地逆向 METAR 生成算法（如验证 5 分钟平均窗口的具体实现），AEMET 10 分钟数据是必要的。对于 Polymarket 交易策略而言，v3 API 的 `temperatureMaxSince7Am` 已经足够。
